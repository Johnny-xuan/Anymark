# 产品定义工作流

把清晰的概念变成可执行的产品设计。

## 产品定义文档模板

### 1. 一句话描述

```
[产品名] 是一个 [产品类型]，
帮助 [目标用户]
解决 [核心问题]，
通过 [核心方式]。
```

**示例：**
- "Notion 是一个协作工具，帮助团队在一个地方组织工作、文档和知识。"
- "Figma 是一个设计工具，让设计师可以在浏览器中实时协作设计。"

```
- 场景：_______________

WHEN - 什么时候使用？
```


## 功能优先级框架

### RICE 评分

```
功能：_______________

Reach（影响范围）：___/10
- 多少用户会用到？

Impact（影响程度）：___/10
- 对用户价值有多大？

Confidence（信心）：___/10
- 我们有多确定？

Effort（工作量）：___/10
- 需要多少开发时间？

RICE 分数 = (R × I × C) / E = ___
```

### MoSCoW 分类

```
Must Have（必须）：没有就不能用
Should Have（应该）：重要但不紧急
Could Have（可以）：锦上添花
Won't Have（不做）：明确不做
```

## 设计原则

### 定义产品原则

```
我们的产品原则：

1. _______________
   示例：简单优先于功能丰富

2. _______________
   示例：速度优先于完美

3. _______________
   示例：用户控制优先于自动化
```

### 设计决策检查

```
每个设计决策都要问：

- [ ] 这解决了用户的核心问题吗？
- [ ] 这符合我们的产品原则吗？
- [ ] 这是 MVP 必须的吗？
- [ ] 用户能理解怎么用吗？
- [ ] 我们能在预期时间内做出来吗？
```

## 验证清单

```
产品定义完成检查：

- [ ] 一句话能说清楚产品是什么
- [ ] 目标用户画像清晰具体
- [ ] 核心问题有数据/调研支撑
- [ ] MVP 功能列表明确且精简
- [ ] 差异化优势清晰
- [ ] 成功指标可衡量
- [ ] 风险已识别并有应对方案
- [ ] 团队对定义达成共识
```
